The Weather Forecasts endpoint is a sample endpoint included in the template to demonstrate how a query flows through the Clean Architecture layers from the HTTP endpoint down to the application handler. It generates five days of random forecast data on each request — it does not connect to any real weather service.

The endpoint exists primarily as a reference for building your own feature slices and can be safely removed once you understand the pattern.

---

## GET /api/WeatherForecasts

Returns an array of five weather forecasts starting from tomorrow, each with a date, temperature in Celsius, a derived temperature in Fahrenheit, and a randomly chosen summary string.

**Authentication:** required  
**Returns:** `200 OK` with an array of `WeatherForecast` objects

### Response shape — WeatherForecast

<ResponseField name="date" type="string (ISO 8601 datetime)">
  The date of the forecast. Always one to five days in the future relative to the time of the request.
</ResponseField>

<ResponseField name="temperatureC" type="integer">
  Temperature in degrees Celsius. Randomly generated in the range −20 to 54.
</ResponseField>

<ResponseField name="temperatureF" type="integer">
  Temperature in degrees Fahrenheit, derived from `temperatureC` using `32 + (int)(temperatureC / 0.5556)`.
</ResponseField>

<ResponseField name="summary" type="string">
  A human-readable description of the forecast. One of: `Freezing`, `Bracing`, `Chilly`, `Cool`, `Mild`, `Warm`, `Balmy`, `Hot`, `Sweltering`, `Scorching`.
</ResponseField>

<Note>
  The data is generated in-memory using `System.Random` and is different on every request. There is no caching, persistence, or external data source involved.
</Note>

## How the endpoint is structured

The endpoint follows the same Clean Architecture pattern used throughout the template:

1. **HTTP layer** — `WeatherForecasts.cs` maps `GET /api/WeatherForecasts` to a handler method that dispatches a `GetWeatherForecastsQuery` via MediatR.
2. **Application layer** — `GetWeatherForecastsQueryHandler` generates the random forecast list and returns it as `IEnumerable<WeatherForecast>`.
3. **No infrastructure dependency** — because the data is synthetic, the handler does not depend on `IApplicationDbContext` or any external service, making it useful as a minimal working example.

This same three-layer flow — minimal API endpoint, MediatR query, handler — is the pattern you replicate when adding new features to the template.
